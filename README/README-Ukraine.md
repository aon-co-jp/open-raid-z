# open-raid-z

Експериментальний проєкт файлової системи для Windows, що зберігає майже повну сумісність із NTFS/exFAT, надаючи функції у стилі ZFS (самовідновлювані контрольні суми, пули зберігання, копіювання при записі, знімки/клони) поряд із RAID0/1/5/6/10/Z2/Z3.

## Угода про іменування

Ідентифікатори, визначені самим проєктом, — імена каталогів, назви crate, імена npm-пакетів, назви features Cargo, id/класи HTML/CSS тощо — послідовно використовують **знак підкреслення (`_`) замість дефіса (`-`)** (наприклад, `open_zfs_winfsp_bridge`, `zfs_accel_hlsl`, `open_runo_installer`, `open_runo_installer_core`, а також features Cargo `winfsp_backend`/`gpu_accel`). Назви, що раніше містили дефіси, як-от `openzfs-winfsp-bridge`, було перейменовано для узгодженості в межах проєкту.

Винятками є такі випадки, оскільки вони підпорядковуються зовнішнім специфікаціям чи угодам екосистеми, а не власному вибору іменування цього проєкту:

- Назва самого репозиторію (`open-raid-z`; це реальна назва репозиторію на GitHub, і її не можна змінити)
- Користувацькі атрибути HTML5 `data-*` (`data-i18n`; дефіс обов'язковий за специфікацією)
- Імена зовнішніх npm-пакетів (наприклад, `@tauri-apps/api` — реальні опубліковані імена пакетів)
- Назви властивостей CSS (наприклад, `font-family` — це сам синтаксис мови CSS)
- Англійські складені терміни, які справді містять дефіс, як-от Reed-Solomon або copy-on-write

## Компоненти

| Компонент | Роль |
|---|---|
| `open_zfs_winfsp_bridge` | vdev'и RAID-Z/RAID0-10, пул зберігання, рівень сумісності ACL NTFS/атрибутів exFAT, справжнє монтування через WinFsp |
| `zfs_accel_hlsl` | Вивантаження обчислення парності на апаратні прискорювачі GPU/NPU (DirectX 12 Compute + DirectML) |
| `open_runo_installer_core` | Незалежна від ОС логіка виявлення дисків, консультанта з конфігурації у стилі Copilot і попереднього перегляду ініціалізації zpool (без залежності від Tauri; `cargo test` працює й на Linux/macOS) |
| `open_runo_installer` | Сам інсталятор на Tauri (тонкий UI-шар, що викликає `open_runo_installer_core`): виявлення обладнання, майстер ініціалізації zpool, інтерфейс консультанта з конфігурації у стилі Copilot |

## Основні можливості

- **Повна серія RAID**: RAID0 / RAID1 (дзеркало) / RAID5 / RAID6 / RAID10 (дзеркала зі стрипінгом) / RAID-Z2 / RAID-Z3
- **Поділ диска на розділи та повторне використання**: розділити один фізичний диск і використати одну половину як учасника дзеркала, а іншу — як учасника окремого масиву RAID6/Z2
- **Самовідновлювані контрольні суми, копіювання при записі, знімки/клони**: імітують підхід ZFS. `Pool::scrub` може за один прохід виявити та усунути приховані пошкодження в усьому пулі через єдиний API як для сімейства RAID-Z, так і для RAID10
- **Сумісність із NTFS**: перетворення ACL (NFSv4 ⇔ NTFS), відповідність UID/GID ⇔ SID (детермінована відповідність на основі RID для локальних доменів SAM/AD)
- **Сумісність з exFAT**: перетворення атрибутів файлів і часових міток, підтримка файлів/томів понад 4 ГБ
- **Апаратне прискорення GPU/NPU**: обчислення парності RAID-Z1/Z2 реально виконується через DirectX 12 Compute + DirectML (автоматичний відкат на CPU за відсутності обладнання)
- **Консультант з конфігурації у стилі Copilot**: рекомендує рівень RAID на основі конфігурації дисків, прискорювача та кількості ядер CPU (перша евристична версія; також реалізовано каркас для виявлення локальної LLM). Логіка міститься в `open_runo_installer_core`, незалежному від Tauri, і її можна перевірити за допомогою `cargo test` також на Linux/macOS
- **Справжнє монтування через WinFsp**: може бути реально змонтований як буква диска Windows. Кожен набір даних у пулі відображається як окремий файл, з підтримкою довільних байтових зміщень під час читання/запису, а також створення/видалення/перейменування/дозапису/усічення файлів (простір імен залишається плоским у корені — підкаталоги поки не підтримуються). Перевірено на реальному обладнанні: читання, запис, створення, видалення, перейменування, дозапис і усічення файлів через реально змонтований диск.
- **Багатомовна підтримка**: інсталятор за замовчуванням використовує японську мову з перемикачем мови інтерфейсу, який можна змінити й після встановлення
- **Інструмент міграції наявних даних (модуль `migrate`, експериментальний)**: копіює наявне дерево каталогів NTFS (або подібної ФС) у пул. Він ніколи не пише в джерело, тому може виконуватися **поки Windows продовжує працювати**. Проте він не може перетворити **на місці поточний робочий системний диск (C: тощо)** у формат RAID без простою (ОС не може допустити, щоб том, який вона активно використовує, переписувався програмою, що працює на ній же, — це принципове обмеження, а не відсутня функція). Це виключно інструмент «копіювання в інше місце (у пул)». Наразі доступний лише як функція бібліотеки, без CLI/GUI; підкаталоги сплощуються в один рівень за допомогою символу-роздільника.

## Обмеження ємності та розміру файлів

- Логічний розмір набору даних (файлу) послідовно відстежується як `u64`, тому штучного обмеження на кшталт межі 4 ГБ у FAT32 немає (теоретична межа — 2^64 байт). Великі файли, як-от відео чи зображення, зберігаються без проблем, поки вкладаються в реальні обмеження нижче.
- Реальна межа — це **вільна ємність пулу**: сума придатної до використання ємності підключених дисків мінус накладні витрати на надлишковість відповідного рівня RAID. Наприклад, для RAID-Z2 (подвійна парність) фактична межа приблизно дорівнює сумарній ємності дисків із даними.
- Один виклик читання/запису WinFsp обмежений приблизно 4 ГіБ (`u32`) самим Windows API, але це те саме обмеження, що й у будь-якій реальній файловій системі, — ОС/застосунок автоматично розбиває більші передавання на кілька викликів, тому практичного обмеження це не створює.
- Через копіювання при записі (CoW) для будь-якого запису (створення, дозапису чи перезапису) завжди потрібна щонайменше одна вільна смуга (stripe) у пулі (та сама ідея, що й `slop space` у ZFS). Заповнення пулу на 100 % означає, що навіть перезапис наявних даних завершиться помилкою. На практиці завжди варто залишати вільними кілька відсотків пулу.

## Поточні обмеження (стадія прототипу)

- Монтування через WinFsp підтримує лише плаский простір імен у корені. Підкаталоги не підтримуються (створення/видалення/перейменування окремих файлів — підтримуються).
- Читання/запис проходять через `Pool::read_unaligned`/`Pool::write_unaligned_growing` (шар read-modify-write) і підтримують довільні байтові зміщення/довжини; запис, що перевищує поточний розмір, автоматично збільшує файл (див. розділ «Обмеження ємності та розміру файлів» вище щодо ємності й налаштування PATH).
- `Pool` підтримує як `RaidZVdev`, так і `Raid10Vdev`, але інтеграція RAID10 з API набору даних місцями ще поверхнева.
- Код справжнього монтування WinFsp (`mount.rs`) не можна зібрати з версією інструментів Rust старішою за 1.85, оскільки crate `winfsp` вимагає функцію Cargo `edition2024` (див. розділ «Збірка та тестування» нижче).
- `mount.rs` і реалізація GPU в `zfs_accel_hlsl` (feature `gpu`) залежать від crate `windows`, вміст якого повністю порожній, якщо метою компіляції не є реально Windows. Тому цей код можна зібрати й протестувати лише на справжній машині з Windows (або під час крос-компіляції під ціль Windows); на Linux/macOS він збирається лише після вимкнення цих функцій через `--no-default-features`.
- Перейменування (`rename`) файлу, коли на нього все ще вказує інший відкритий дескриптор, може призвести до того, що цей інший дескриптор перестане працювати для подальших операцій (`FileHandle` за задумом зберігає ім'я безпосередньо — подробиці див. у документації `Pool::rename_dataset`).

## Збірка та тестування

```powershell
cd open_runo_zfs_source/open_zfs_winfsp_bridge
cargo test --no-default-features   # без монтування WinFsp і прискорення GPU (лише логіка CPU; не потрібні ні dxc, ні SDK WinFsp)
cargo test                         # за замовчуванням (включає справжнє монтування WinFsp та прискорення GPU/NPU; потрібні WinFsp + dxc)
```

`--no-default-features` вимикає обидві функції `winfsp_backend` і `gpu_accel`, дозволяючи перевірити основну логіку — RAID0/1/5/6/10/Z2/Z3, самовідновлювані контрольні суми, CoW, знімки/клони, resilver тощо — незалежно від ОС (працює й на Linux/macOS). Не потрібні ні WinFsp, ні DirectX Shader Compiler (dxc), ні обладнання GPU/NPU.

Збірка зі стандартними функціями (`winfsp_backend` + `gpu_accel`) вимагає:

- Установленого в системі середовища виконання WinFsp (https://winfsp.dev/) (заголовки SDK, що використовуються під час збірки, постачаються автоматично, тому окреме встановлення компонента для розробників не потрібне).
- `dxc` (DirectX Shader Compiler, входить до складу Windows SDK або Vulkan SDK) у `PATH` (використовується для компіляції HLSL-шейдерів парності RAID-Z/Z2 під час збірки).
- **Rust 1.85 або новішої версії** (версія, у якій було стабілізовано `edition2024`, потрібну crate `winfsp`; на старіших інструментах уже не вдається розібрати маніфест `Cargo.toml`).

WinFsp і dxc також можна вимкнути окремо (наприклад, `--no-default-features --features gpu_accel` — лише GPU, без WinFsp).

**Примітка щодо реального запуску тестів `winfsp_backend` (справжнє монтування)**: crate `winfsp` динамічно завантажує DLL WinFsp (`winfsp-x64.dll`) через `LoadLibraryW`, яка шукає лише за стандартним шляхом пошуку DLL (папка виконуваного файлу, `System32` і `PATH`). У середовищах, де інсталятор WinFsp не додав себе до `PATH`, збірка проходить успішно (заголовки SDK WinFsp не потрібні), але запуск **завжди завершується помилкою під час виконання** (помилка `WIN32(1285)` = `ERROR_DELAY_LOAD_FAILED`). Додайте каталог `bin` WinFsp до `PATH` лише на час запуску тестів:

```powershell
$env:PATH = "C:\Program Files (x86)\WinFsp\bin;$env:PATH"
cargo test --features winfsp_backend,gpu_accel
```

Без цього `mount_pool` повертає `Err`, і тест трактує це як проблему, залежну від середовища, виводячи через `eprintln` повідомлення про пропуск і завершуючись достроково. **Без `--nocapture` цей пропуск усе одно відображається просто як `ok`, невідрізнимо від справді успішного монтування й читання/запису.** Під час перевірки тестів справжнього монтування завжди додавайте `--nocapture` і візуально переконуйтеся, що повідомлення про пропуск не з'являється.

### Інсталятор (`open_runo_installer` / `open_runo_installer_core`)

```powershell
# Шар логіки (без залежності від Tauri; також працює на Linux/macOS)
cd open_runo_zfs_source/open_runo_installer_core
cargo test                    # лише відкат на CPU (за замовчуванням)
cargo test --features gpu     # включно зі справжньою диспетчеризацією GPU/NPU (потрібна справжня машина з Windows + dxc)

# Фронтенд (TypeScript, незалежний від ОС)
cd open_runo_zfs_source/open_runo_installer
npm install
npx tsc --noEmit               # лише перевірка типів
npx vite build                 # реальна збірка

# Сам застосунок Tauri (потрібна справжня машина з Windows або достатньо новий Rust плюс залежності для стільниці Linux)
cd open_runo_zfs_source/open_runo_installer/src-tauri
cargo tauri dev / cargo tauri build
```

`open_runo_installer_core` (виявлення дисків, консультант з конфігурації у стилі Copilot, попередній перегляд ініціалізації zpool) — незалежний crate без залежності від Tauri, тому його логіку можна перевірити як є навіть у середовищах, де відсутнє все необхідне самому Tauri для збірки (WebView, GTK тощо, а також достатньо нова версія інструментів Rust). Лише реальне перелічення дисків (`\\.\PhysicalDriveN`) використовує специфічний для Windows API та ізольоване за `#[cfg(windows)]`; усе інше (консультант з конфігурації та розрахунки попереднього перегляду zpool) не залежить від ОС, і всі 26 його тестів підтверджено проходять.
