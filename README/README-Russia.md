# open-raid-z

Экспериментальный проект файловой системы для Windows/Linux, сохраняющий почти полную совместимость с NTFS/exFAT и предоставляющий функции в стиле ZFS (самовосстанавливающиеся контрольные суммы, пулы хранения, копирование при записи, снимки/клоны) наряду с RAID0/1/5/6/10/Z2/Z3. Основная логика — единая, не зависящая от ОС общая программа (`open_raid_z_core`); версия для Windows (WinFsp) и версия для Linux (FUSE) отличаются лишь тонким слоем монтирования поверх неё (распространяются под именами `open-raid-z-win`/`open-raid-z-linux`).

## Обращение к Microsoft и Apple

Мы разрабатываем эту экспериментальную файловую систему, чтобы привнести в Windows полноценные возможности в стиле ZFS (самовосстанавливающиеся контрольные суммы, RAID6/RAID-Z2, снимки и многое другое). Одна из наших долгосрочных целей — чтобы эту файловую систему когда-нибудь можно было официально выбрать как цель установки и системный диск в Windows и macOS.

Мы понимаем, что для этого требуется содействие со стороны каждого производителя ОС — подпись/сертификация загрузочных драйверов, официальная поддержка в установщике и так далее. Если у вас есть интерес к этой работе, мы будем очень рады обращению и сотрудничеству с вашей стороны. Это небольшой независимый проект, но мы искренне стремимся довести эту технологию до реализации.

## Соглашение об именовании

Идентификаторы, определяемые самим проектом, — имена каталогов, названия crate, имена npm-пакетов, названия features Cargo, id/классы HTML/CSS и т. д. — последовательно используют **знак подчёркивания (`_`) вместо дефиса (`-`)** (например, `open_raid_z_core`, `zfs_accel_hlsl`, `open_runo_installer`, `open_runo_installer_core`, а также features Cargo `winfsp_backend`/`gpu_accel`). Названия, ранее содержавшие дефисы, такие как `openzfs-winfsp-bridge`, были переименованы для обеспечения единообразия внутри проекта.

Исключение составляют следующие случаи, поскольку они следуют внешним спецификациям или соглашениям экосистемы, а не собственному выбору именования этого проекта:

- Название самого репозитория (`open-raid-z`; это реальное имя репозитория на GitHub, и его нельзя изменить)
- Пользовательские атрибуты HTML5 `data-*` (`data-i18n`; дефис обязателен по спецификации)
- Имена внешних npm-пакетов (например, `@tauri-apps/api` — реальные опубликованные имена пакетов)
- Названия свойств CSS (например, `font-family` — это сам синтаксис языка CSS)
- Английские составные термины, действительно содержащие дефис, такие как Reed-Solomon или copy-on-write

## Компоненты

| Компонент | Роль |
|---|---|
| `open_raid_z_core` | vdev'ы RAID-Z/RAID0-10, пул хранения, слой совместимости ACL NTFS/атрибутов exFAT, настоящее монтирование (Windows = WinFsp `mount.rs` / Linux = FUSE `fuse_mount.rs`; всё, кроме специфичного для ОС слоя монтирования, полностью общее) |
| `zfs_accel_hlsl` | Разгрузка вычисления четности на аппаратные ускорители GPU/NPU (DirectX 12 Compute + DirectML) |
| `open_runo_installer_core` | Не зависящая от ОС логика обнаружения дисков, консультанта по конфигурации в стиле Copilot и предварительного просмотра инициализации zpool (без зависимости от Tauri; `cargo test` работает и на Linux/macOS) |
| `open_runo_installer` | Сам установщик на Tauri (тонкий UI-слой, вызывающий `open_runo_installer_core`): обнаружение оборудования, мастер инициализации zpool, интерфейс консультанта по конфигурации в стиле Copilot |

## Основные возможности

- **Полная серия RAID**: RAID0 / RAID1 (зеркало) / RAID5 / RAID6 / RAID10 (чередующиеся зеркала) / RAID-Z2 / RAID-Z3
- **Разбиение диска на разделы и повторное использование**: разделить один физический диск и использовать одну половину как участник зеркала, а другую — как участник отдельного массива RAID6/Z2
- **Самовосстанавливающиеся контрольные суммы, копирование при записи, снимки/клоны**: имитируют подход ZFS. `Pool::scrub` может за один проход обнаружить и устранить скрытые повреждения во всём пуле через единый API как для семейства RAID-Z, так и для RAID10
- **Совместимость с NTFS**: преобразование ACL (NFSv4 ⇔ NTFS), сопоставление UID/GID ⇔ SID (детерминированное сопоставление на основе RID для локальных доменов SAM/AD)
- **Совместимость с exFAT**: преобразование атрибутов файлов и меток времени, поддержка файлов/томов объёмом более 4 ГБ
- **Аппаратное ускорение GPU/NPU**: генерация четности RAID-Z1/Z2/Z3 реально выполняется через DirectX 12 Compute + DirectML (автоматический откат на CPU при отсутствии оборудования). Также реализована схема, преобразующая умножение на коэффициент GF(2^8) в битовую матрицу GF(2) и сводящая его к одному вызову DirectML GEMM (`zfs_accel_hlsl::dml_gemm`), корректность которой подтверждена на реальном GPU (на реальном NPU пока не проверена). Этот же механизм задействован в вычислении восстановления, которое выполняют scrub/resilver при обнаружении повреждения (то есть в проверке четности). Также подготовлены отдельные шейдерные пути для NPU (`raidnpu_*.hlsl`) — на будущее, для проверки/оптимизации на реальном NPU
- **Применение zpool к реальным дискам**: в мастере инициализации zpool установщика теперь есть команда (`init_zpool_apply`), применяющая изменения к реальным физическим дискам (`\\.\PhysicalDriveN`), а не только к предпросмотру на временном образе. Защищена явным флагом подтверждения удаления существующих данных
- **Консультант по конфигурации в стиле Copilot**: рекомендует уровень RAID на основе конфигурации дисков, ускорителя и количества ядер CPU (первая эвристическая версия; также реализован каркас для обнаружения локальной LLM). Логика находится в `open_runo_installer_core`, независимом от Tauri, и её можно проверить с помощью `cargo test` также на Linux/macOS
- **Настоящее монтирование через WinFsp (Windows)**: может быть реально смонтирован как буква диска Windows. Каждый набор данных в пуле отображается как отдельный файл, с поддержкой произвольных байтовых смещений при чтении/записи, а также создания/удаления/переименования/дозаписи/усечения файлов (пространство имён остаётся плоским в корне — подкаталоги пока не поддерживаются). Проверено на реальном оборудовании: чтение, запись, создание, удаление, переименование, дозапись и усечение файлов через реально смонтированный диск.
- **Настоящее монтирование через FUSE (Linux)**: тот же самый `Pool` монтируется напрямую и в Linux (`fuse_mount.rs`), с тем же функционалом, что и в версии для Windows (создание/удаление/переименование/дозапись/усечение). Проверено полностью на WSL2 Ubuntu 26.04 — смонтировано и опробовано через обычные вызовы `std::fs`. Поскольку реализация основана на inode, ей не свойственно известное ограничение версии WinFsp, при котором другой открытый дескриптор может продолжать ссылаться на устаревшее имя после переименования. В crate `fuser` также есть feature `macfuse-4-compat`, поэтому та же архитектура в перспективе может быть расширена на macOS (как том данных, а не диск загрузки).
- **Многоязычная поддержка**: установщик (OpenRaidZ Installer) по умолчанию использует английский язык с переключателем языка интерфейса (10 языков), который можно изменить и после установки
- **Инструмент миграции существующих данных (модуль `migrate`, экспериментальный)**: копирует существующее дерево каталогов NTFS (или аналогичной ФС) в пул. Он никогда не пишет в источник, поэтому может выполняться **пока Windows продолжает работать**. Однако он не может преобразовать **на месте текущий работающий системный диск (C: и т. д.)** в формат RAID без простоя (ОС не может допустить, чтобы том, который она активно использует, переписывался программой, работающей на ней же, — это принципиальное ограничение, а не недостающая функция). Это строго инструмент «копирования в другое место (в пул)». Пока доступен только как функция библиотеки, без CLI/GUI; подкаталоги сплющиваются в один уровень с помощью символа-разделителя.
- **Постоянное хранение метаданных (`Pool::save`/`Pool::open`)**: список наборов данных, распределение полос, снимки и другую служебную информацию теперь можно сохранять в зарезервированную область (суперблок) внутри пула и восстанавливать из неё. Раньше такого механизма не было — необработанные байты данных сохранялись на диске, но запись о том, где находился каждый файл, терялась, как только процесс (монтирование) завершался. И версия для Windows (WinFsp), и версия для Linux (FUSE) теперь автоматически сохраняют состояние при каждой изменяющей операции, и на реальном оборудовании подтверждено, что файлы действительно сохраняются после настоящего размонтирования и повторного монтирования.

## Ограничения ёмкости и размера файлов

- Логический размер набора данных (файла) последовательно отслеживается как `u64`, поэтому искусственного ограничения вроде границы 4 ГБ у FAT32 не существует (теоретический предел — 2^64 байт). Большие файлы, такие как видео или изображения, сохраняются без проблем, пока укладываются в реальные ограничения ниже.
- Реальный предел — это **свободная ёмкость пула**: сумма используемой ёмкости подключённых дисков за вычетом накладных расходов на избыточность соответствующего уровня RAID. Например, при RAID-Z2 (двойная четность) фактический предел примерно равен суммарной ёмкости дисков с данными.
- Один вызов чтения/записи WinFsp ограничен примерно 4 ГиБ (`u32`) самим Windows API, но это то же самое ограничение, что и у любой реальной файловой системы, — ОС/приложение автоматически разбивает более крупные передачи на несколько вызовов, поэтому практического ограничения это не создаёт.
- Из-за копирования при записи (CoW) для любой записи (создания, дозаписи или перезаписи) всегда требуется как минимум одна свободная полоса (stripe) в пуле (та же идея, что и `slop space` в ZFS). Ещё одна полоса постоянно зарезервирована для хранения метаданных. Заполнение пула на 100 % означает, что даже перезапись существующих данных завершится ошибкой. На практике всегда следует оставлять свободными несколько процентов пула.

## Текущие ограничения (стадия прототипа)

- Монтирование через WinFsp поддерживает только плоское пространство имён в корне. Подкаталоги не поддерживаются (создание/удаление/переименование отдельных файлов — поддерживаются).
- Чтение/запись проходят через `Pool::read_unaligned`/`Pool::write_unaligned_growing` (слой read-modify-write) и поддерживают произвольные байтовые смещения/длины; запись, превышающая текущий размер, автоматически увеличивает файл (см. раздел «Ограничения ёмкости и размера файлов» выше о ёмкости и настройке PATH).
- `Pool` поддерживает как `RaidZVdev`, так и `Raid10Vdev`, но интеграция RAID10 с API набора данных местами всё ещё поверхностна.
- Код настоящего монтирования WinFsp (`mount.rs`) нельзя собрать с версией инструментов Rust младше 1.85, поскольку crate `winfsp` требует функцию Cargo `edition2024` (см. раздел «Сборка и тестирование» ниже).
- `mount.rs` и реализация GPU в `zfs_accel_hlsl` (feature `gpu`) зависят от crate `windows`, содержимое которого полностью пусто, если целью компиляции не является реально Windows. Поэтому этот код можно собрать и протестировать только на настоящей машине с Windows (либо при кросс-компиляции под цель Windows); на Linux/macOS он собирается только при отключении этих функций через `--no-default-features`.
- Переименование (`rename`) файла при наличии другого открытого дескриптора, всё ещё указывающего на него, может привести к тому, что этот другой дескриптор перестанет работать для последующих операций (`FileHandle` по конструкции хранит имя напрямую — подробности см. в документации `Pool::rename_dataset`).

## Сборка и тестирование

```powershell
cd open_runo_zfs_source/open_raid_z_core
cargo test --no-default-features   # без монтирования WinFsp и ускорения GPU (только логика CPU; не нужны ни dxc, ни SDK WinFsp)
cargo test                         # по умолчанию (включает настоящее монтирование WinFsp и ускорение GPU/NPU; требуется WinFsp + dxc)
```

`--no-default-features` отключает обе функции `winfsp_backend` и `gpu_accel`, позволяя проверить основную логику — RAID0/1/5/6/10/Z2/Z3, самовосстанавливающиеся контрольные суммы, CoW, снимки/клоны, resilver и т. д. — независимо от ОС (работает и на Linux/macOS). Не требуются ни WinFsp, ни DirectX Shader Compiler (dxc), ни оборудование GPU/NPU.

Сборка с функциями по умолчанию (`winfsp_backend` + `gpu_accel`) требует:

- Установленной в системе среды выполнения WinFsp (https://winfsp.dev/) (заголовки SDK, используемые при сборке, поставляются автоматически, поэтому отдельная установка компонента для разработчиков не требуется).
- `dxc` (DirectX Shader Compiler, входит в состав Windows SDK или Vulkan SDK) в `PATH` (используется для компиляции HLSL-шейдеров четности RAID-Z/Z2 во время сборки).
- **Rust 1.85 или новее** (версия, в которой была стабилизирована `edition2024`, требуемая crate `winfsp`; на более старых инструментах уже не удаётся разобрать манифест `Cargo.toml`).

WinFsp и dxc также можно отключить по отдельности (например, `--no-default-features --features gpu_accel` — только GPU, без WinFsp).

**Замечание по реальному запуску тестов `winfsp_backend` (настоящее монтирование)**: crate `winfsp` динамически загружает DLL WinFsp (`winfsp-x64.dll`) через `LoadLibraryW`, которая ищет только по стандартному пути поиска DLL (папка исполняемого файла, `System32` и `PATH`). В окружениях, где установщик WinFsp не добавил себя в `PATH`, сборка проходит успешно (заголовки SDK WinFsp не требуются), но запуск **всегда завершается ошибкой во время выполнения** (ошибка `WIN32(1285)` = `ERROR_DELAY_LOAD_FAILED`). Добавьте каталог `bin` WinFsp в `PATH` только на время запуска тестов:

```powershell
$env:PATH = "C:\Program Files (x86)\WinFsp\bin;$env:PATH"
cargo test --features winfsp_backend,gpu_accel
```

Без этого `mount_pool` возвращает `Err`, и тест трактует это как проблему, зависящую от окружения, выводя через `eprintln` сообщение о пропуске и завершаясь досрочно. **Без `--nocapture` этот пропуск всё равно отображается просто как `ok`, неотличимо от реально успешного монтирования и чтения/записи.** При проверке тестов настоящего монтирования всегда указывайте `--nocapture` и визуально убеждайтесь, что сообщение о пропуске не появляется.

### Сборка и тестирование версии для Linux (FUSE)

```bash
# Для Ubuntu/Debian требуются build-essential, pkg-config и libfuse3-dev.
sudo apt-get install -y build-essential pkg-config libfuse3-dev

cd open_runo_zfs_source/open_raid_z_core
cargo test --no-default-features --features fuse_backend
```

Функция `fuse_backend` включает crate `fuser` (настоящую привязку к `libfuse3` в Linux). Она независима от `winfsp_backend`/`gpu_accel` и не может быть включена на платформах, отличных от Linux, поскольку сам `fuser` там даже не является зависимостью (он находится в `Cargo.toml` под `target.'cfg(target_os = "linux")'.dependencies`). Интеграционный тест настоящего монтирования (`tests/fuse_mount.rs`) проверен на WSL2 Ubuntu 26.04 — создание, запись, чтение, переименование, усечение, удаление, цикл чтения-записи для файла большего размера, охватывающего несколько полос, и сохранение метаданных после настоящего размонтирования и повторного монтирования. Если вы работаете только на Windows, для сборки/тестирования Linux-цели рекомендуется WSL2 (`wsl --install`).

Также включена небольшая утилита командной строки `orzctl` для создания и монтирования пула прямо из терминала:

```bash
cargo build --no-default-features --features fuse_backend --bin orzctl
./target/debug/orzctl create --level z2 --chunk-size 4096 --stripes 1000 --dataset tank /path/to/disk0 /path/to/disk1 ...
./target/debug/orzctl mount  --level z2 --chunk-size 4096 --stripes 1000 --mountpoint /mnt/tank /path/to/disk0 /path/to/disk1 ...
```

### Установщик (`open_runo_installer` / `open_runo_installer_core`)

```powershell
# Слой логики (без зависимости от Tauri; также работает на Linux/macOS)
cd open_runo_zfs_source/open_runo_installer_core
cargo test                    # только откат на CPU (по умолчанию)
cargo test --features gpu     # включая настоящую диспетчеризацию GPU/NPU (требуется настоящая машина с Windows + dxc)

# Фронтенд (TypeScript, независимый от ОС)
cd open_runo_zfs_source/open_runo_installer
npm install
npx tsc --noEmit               # только проверка типов
npx vite build                 # реальная сборка

# Само приложение Tauri (требуется настоящая машина с Windows либо достаточно новый Rust плюс зависимости для рабочего стола Linux)
cd open_runo_zfs_source/open_runo_installer/src-tauri
cargo tauri dev / cargo tauri build
```

`open_runo_installer_core` (обнаружение дисков, консультант по конфигурации в стиле Copilot, предварительный просмотр инициализации zpool) — независимый crate без зависимости от Tauri, поэтому его логику можно проверить как есть даже в окружениях, где отсутствует всё необходимое самому Tauri для сборки (WebView, GTK и т. д., а также достаточно новая версия инструментов Rust). Только реальное перечисление дисков (`\\.\PhysicalDriveN`) использует API, специфичный для Windows, и изолировано за `#[cfg(windows)]`; всё остальное (консультант по конфигурации и расчёты предпросмотра zpool) не зависит от ОС, и все 26 его тестов подтверждённо проходят.
