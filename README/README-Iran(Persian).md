# open-raid-z

یک پروژه‌ی آزمایشی سیستم فایل برای ویندوز/لینوکس که سازگاری تقریباً کامل با NTFS/exFAT را حفظ می‌کند و ویژگی‌هایی به سبک ZFS (چک‌سام‌های خودترمیم‌شونده، استخرهای ذخیره‌سازی، کپی هنگام نوشتن، عکس‌های لحظه‌ای/کلون‌ها) را در کنار RAID0/1/5/6/10/Z2/Z3 ارائه می‌دهد. منطق اصلی یک برنامه‌ی مشترکِ مستقل از سیستم‌عامل است (`open_raid_z_core`)؛ نسخه‌ی ویندوز (WinFsp) و نسخه‌ی لینوکس (FUSE) فقط در لایه‌ی نازک سوارکردن روی آن تفاوت دارند (با نام‌های `open-raid-z-win`/`open-raid-z-linux` توزیع می‌شوند).

## پیامی برای Microsoft و Apple

ما این سیستم فایل آزمایشی را توسعه می‌دهیم تا قابلیت‌های کامل به سبک ZFS (چک‌سام‌های خودترمیم‌شونده، RAID6/RAID-Z2، عکس‌های لحظه‌ای و موارد دیگر) را به ویندوز بیاوریم. یکی از اهداف بلندمدت ما این است که این سیستم فایل روزی بتواند به‌عنوان یک مقصد نصب و درایو راه‌اندازی رسمی در ویندوز و macOS انتخاب شود.

می‌دانیم که این کار به همکاری هر سازنده‌ی سیستم‌عامل نیاز دارد — امضا/گواهی‌نامه‌ی درایورهای راه‌اندازی، پشتیبانی رسمی در نصب‌کننده و مانند آن‌ها. اگر به این تلاش علاقه‌مند هستید، از تماس و همکاری شما بسیار استقبال می‌کنیم. این یک پروژه‌ی کوچک و مستقل است، اما ما صادقانه مصمم به تحقق این فناوری هستیم.

## قاعده‌ی نام‌گذاری

شناسه‌هایی که خود این پروژه تعریف می‌کند — نام پوشه‌ها، نام crateها، نام بسته‌های npm، نام featureهای Cargo، شناسه‌ها/کلاس‌های HTML/CSS و مانند آن‌ها — به‌طور یکدست از **خط زیرین (`_`) به‌جای خط تیره (`-`)** استفاده می‌کنند (برای نمونه `open_raid_z_core`، `zfs_accel_hlsl`، `open_runo_installer`، `open_runo_installer_core`، و featureهای Cargo مانند `winfsp_backend`/`gpu_accel`). نام‌هایی که پیش‌تر با خط تیره نوشته می‌شدند، مانند `openzfs-winfsp-bridge`، برای یکدستی در سراسر پروژه تغییر نام یافته‌اند.

موارد زیر از این قاعده مستثنا هستند، زیرا از مشخصات بیرونی یا قراردادهای اکوسیستم پیروی می‌کنند، نه از انتخاب نام‌گذاری خود این پروژه:

- نام خود مخزن (`open-raid-z`؛ این نام واقعی مخزن در گیت‌هاب است و قابل تغییر نیست)
- ویژگی‌های سفارشی HTML5 از نوع `data-*` (مانند `data-i18n`؛ خط تیره طبق مشخصات الزامی است)
- نام بسته‌های npm بیرونی (مانند `@tauri-apps/api`، که نام واقعی بسته‌های منتشرشده هستند)
- نام ویژگی‌های CSS (مانند `font-family`؛ این خود نحو زبان CSS است)
- اصطلاحات مرکب انگلیسی که واقعاً دارای خط تیره هستند، مانند Reed-Solomon یا copy-on-write

## اجزا

| جزء | نقش |
|---|---|
| `open_raid_z_core` | vdevهای RAID-Z/RAID0-10، استخر ذخیره‌سازی، لایه‌ی سازگاری ACL در NTFS/ویژگی‌های exFAT، سوارکردن واقعی (ویندوز = WinFsp، `mount.rs` / لینوکس = FUSE، `fuse_mount.rs`؛ همه‌چیز به‌جز لایه‌ی سوارکردن مخصوص هر سیستم‌عامل کاملاً مشترک است) |
| `zfs_accel_hlsl` | واگذاری محاسبه‌ی توازن (parity) به شتاب‌دهنده‌های سخت‌افزاری GPU/NPU (DirectX 12 Compute + DirectML) |
| `open_runo_installer_core` | منطق مستقل از سیستم‌عامل برای شناسایی دیسک، مشاور پیکربندی به سبک Copilot و پیش‌نمایش مقداردهی اولیه‌ی zpool (بدون هیچ وابستگی به Tauri؛ `cargo test` روی Linux/macOS نیز کار می‌کند) |
| `open_runo_installer` | خودِ نصب‌کننده‌ی مبتنی بر Tauri (یک لایه‌ی رابط کاربری نازک که `open_runo_installer_core` را فرامی‌خواند): شناسایی سخت‌افزار، جادوگر مقداردهی اولیه‌ی zpool، رابط مشاور پیکربندی به سبک Copilot |

## ویژگی‌های کلیدی

- **پشتیبانی کامل از سری RAID**: RAID0 / RAID1 (آینه‌ای) / RAID5 / RAID6 / RAID10 (آینه‌های نواری) / RAID-Z2 / RAID-Z3
- **تقسیم‌بندی و استفاده‌ی مجدد از دیسک**: تقسیم یک دیسک فیزیکی و استفاده از نیمی از آن به‌عنوان عضو یک آرایه‌ی آینه‌ای، در حالی که نیمه‌ی دیگر به یک آرایه‌ی جداگانه‌ی RAID6/Z2 می‌پیوندد
- **چک‌سام‌های خودترمیم‌شونده، کپی هنگام نوشتن، عکس‌های لحظه‌ای/کلون‌ها**: شبیه‌سازی رویکرد ZFS. `Pool::scrub` می‌تواند در یک مرحله، خرابی‌های پنهان را در سراسر استخر شناسایی و ترمیم کند، از طریق یک API یکسان چه برای خانواده‌ی RAID-Z و چه برای RAID10
- **سازگاری با NTFS**: ترجمه‌ی ACL (NFSv4 ⇔ NTFS)، نگاشت UID/GID ⇔ SID (نگاشت قطعی مبتنی بر RID برای دامنه‌های محلی SAM/AD)
- **سازگاری با exFAT**: تبدیل ویژگی‌های فایل و برچسب‌های زمانی، پشتیبانی از فایل‌ها/حجم‌های بزرگ‌تر از ۴ گیگابایت
- **شتاب‌دهی سخت‌افزاری GPU/NPU**: تولید توازن RAID-Z1/Z2/Z3 از طریق DirectX 12 Compute + DirectML اجرا می‌شود (در صورت نبود سخت‌افزار، به‌طور خودکار به CPU بازمی‌گردد). همچنین طرحی پیاده‌سازی شده که ضرب ضریب GF(2^8) را به یک ماتریس بیتی GF(2) تبدیل می‌کند و آن را به یک فراخوانی GEMM در DirectML کاهش می‌دهد (`zfs_accel_hlsl::dml_gemm`)، که درستی آن روی سخت‌افزار GPU واقعی تأیید شده است (روی سخت‌افزار NPU واقعی هنوز تأیید نشده). همین سازوکار به محاسبه‌ی بازسازی که scrub/resilver هنگام تشخیص خرابی اجرا می‌کند (یعنی بررسی توازن) نیز متصل شده است. مسیرهای شیدر مخصوص NPU (`raidnpu_*.hlsl`) نیز فراهم شده تا برای بررسی/بهینه‌سازی آینده روی سخت‌افزار NPU واقعی آماده باشد
- **شتاب‌دهی Vulkan Compute برای پلتفرم‌های غیر ویندوز**: از آنجا که DirectX/DirectML یک API مخصوص ویندوز است، برای پیاده‌سازی همین قابلیت روی Linux/Mac/Android یک نسخه‌ی Vulkan Compute از طریق کریت `ash` اضافه شده است (`zfs_accel_hlsl::vulkan_compute`، ویژگی `vulkan`). درستی تولید توازن XOR در RAID-Z1 روی سخت‌افزار GPU واقعی (NVIDIA GeForce GT 730، Vulkan 1.2) تأیید شده است
- **پل خواندن/نوشتن فرمت‌های خارجی (`foreign_fs`)**: جدا از فرمت اختصاصی استخر open-raid-z، این بخش می‌تواند حجم‌های FAT32/FAT16 و exFAT موجودی که توسط سایر سیستم‌عامل‌ها ایجاد شده‌اند (فلش‌های USB، microSD، کارت‌های CF و مانند آن‌ها) را بخواند و بنویسد — نوشتن روی exFAT هم اکنون با استفاده از کریت `hadris-fat` کار می‌کند. از طریق `orzctl foreign` (`ls`/`cat`/`put`) قابل استفاده است
- **پنل «وضعیت سازگاری» در نصب‌کننده**: با یک دکمه باز/بسته می‌شود و وضعیت پشتیبانی سیستم‌عامل فعلی، هر GPU/NPU شناسایی‌شده (با تشخیص سازنده: Intel/AMD/NVIDIA/Qualcomm، با پشتیبانی از چند دستگاه هم‌زمان) و نوع رسانه‌ی ذخیره‌سازی شناسایی‌شده (HDD/SSD/NVMe/USB/SD/CF) را نشان می‌دهد
- **اعمال zpool روی دیسک واقعی**: ویزارد مقداردهی اولیه‌ی zpool در نصب‌کننده اکنون دستوری دارد (`init_zpool_apply`) که روی دیسک‌های فیزیکی واقعی (`\\.\PhysicalDriveN`) اعمال می‌شود، نه فقط پیش‌نمایش روی تصویر موقت. با پرچم تأیید صریح برای پاک شدن داده‌های موجود محافظت می‌شود
- **مشاور پیکربندی به سبک Copilot**: بر اساس چیدمان دیسک‌ها، شتاب‌دهنده و تعداد هسته‌های CPU، سطح RAID مناسب را پیشنهاد می‌دهد (نسخه‌ی اولیه‌ی اکتشافی؛ یک ساختار پایه برای شناسایی مدل زبانی محلی نیز فراهم شده است). منطق آن در `open_runo_installer_core` قرار دارد که مستقل از Tauri است و می‌توان آن را با `cargo test` روی Linux/macOS نیز بررسی کرد
- **سوارکردن واقعی از طریق WinFsp (ویندوز)**: می‌تواند واقعاً به‌عنوان یک حرف درایو در ویندوز سوار شود. هر مجموعه‌داده در استخر به‌صورت یک فایل مستقل نمایش داده می‌شود و از افست‌های دلخواه بر حسب بایت برای خواندن/نوشتن، و همچنین ایجاد/حذف/تغییرنام/الحاق/کوتاه‌کردن فایل‌ها پشتیبانی می‌شود (فضای نام همچنان در ریشه مسطح است — زیرپوشه‌ها هنوز پشتیبانی نمی‌شوند). روی سخت‌افزار واقعی تأیید شده است: خواندن، نوشتن، ایجاد، حذف، تغییرنام، الحاق و کوتاه‌کردن فایل‌ها از طریق یک درایو واقعاً سوارشده.
- **سوارکردن واقعی از طریق FUSE (لینوکس)**: همان `Pool` مستقیماً روی لینوکس هم سوار می‌شود (`fuse_mount.rs`)، با همان قابلیت‌های نسخه‌ی ویندوز (ایجاد/حذف/تغییرنام/الحاق/کوتاه‌کردن). به‌طور کامل روی WSL2 Ubuntu 26.04 تأیید شده — سوار و از طریق فراخوانی‌های معمولی `std::fs` آزموده شده است. از آنجا که مبتنی بر inode است، محدودیت شناخته‌شده‌ی نسخه‌ی WinFsp (که در آن یک هندل باز دیگر می‌تواند پس از تغییرنام همچنان به نام قدیمی اشاره کند) را ندارد. crate با نام `fuser` همچنین دارای ویژگی `macfuse-4-compat` است، بنابراین همین طراحی احتمالاً در آینده قابل گسترش به macOS خواهد بود (به‌عنوان یک درایو داده، نه دیسک راه‌اندازی).
- **پشتیبانی چندزبانه**: نصب‌کننده (OpenRaidZ Installer) به‌طور پیش‌فرض از انگلیسی استفاده می‌کند و دارای یک تعویض‌کننده‌ی زبان (۹ زبان: انگلیسی، ژاپنی، ایتالیایی، فرانسوی، آلمانی، روسی، اوکراینی، عربی، فارسی) در رابط کاربری است که پس از نصب نیز قابل تغییر است. نمایش ترکیبی (زبان اصلی + یک زبان دوم قابل انتخاب، به‌طور پیش‌فرض ژاپنی) به‌طور پیش‌فرض فعال است و هر دو زبان را در کنار هم نمایش می‌دهد
- **ابزار انتقال داده‌های موجود (ماژول `migrate`، آزمایشی)**: یک درخت پوشه‌ی NTFS (یا مشابه) موجود را به استخر کپی می‌کند. هرگز در مبدأ چیزی نمی‌نویسد، بنابراین می‌تواند **درحالی‌که ویندوز همچنان در حال اجراست** انجام شود. با این حال نمی‌تواند **درایو سیستمی در حال اجرای فعلی (C: و مانند آن) را در همان‌جا** بدون توقف به قالب RAID تبدیل کند (سیستم‌عامل نمی‌تواند اجازه دهد جلدی که هم‌اکنون فعالانه استفاده می‌کند، توسط نرم‌افزاری که خودش روی همان جلد اجرا می‌شود بازنویسی شود — این یک محدودیت بنیادین است، نه یک ویژگی ناقص). این ابزار دقیقاً «کپی به مکانی دیگر (استخر)» است. در حال حاضر فقط یک تابع کتابخانه‌ای است و هنوز CLI/GUI ندارد؛ زیرپوشه‌ها با استفاده از یک نویسه‌ی جداکننده به یک سطح مسطح می‌شوند.
- **پایداری فراداده (`Pool::save`/`Pool::open`)**: اکنون فهرست مجموعه‌داده‌ها، تخصیص نوارها، عکس‌های لحظه‌ای و دیگر اطلاعات مدیریتی را می‌توان در یک ناحیه‌ی رزرو‌شده (سوپربلاک) درون استخر ذخیره و از آن بازیابی کرد. پیش‌تر چنین سازوکاری وجود نداشت — بایت‌های خام داده روی دیسک باقی می‌ماندند، اما رکورد اینکه هر فایل کجا بوده به‌محض پایان فرایند (سوارکردن) از بین می‌رفت. اکنون هم نسخه‌ی ویندوز (WinFsp) و هم نسخه‌ی لینوکس (FUSE) در هر عملیات تغییردهنده به‌طور خودکار ذخیره می‌کنند، و روی سخت‌افزار واقعی تأیید شده که فایل‌ها واقعاً پس از یک پیاده‌سازی و سوارکردن مجدد واقعی باقی می‌مانند.

## ظرفیت و محدودیت‌های اندازه‌ی فایل

- اندازه‌ی منطقی یک مجموعه‌داده (فایل) به‌طور یکدست به‌صورت `u64` پیگیری می‌شود، بنابراین محدودیت مصنوعی مانند مرز ۴ گیگابایتی FAT32 وجود ندارد (سقف نظری ۲^۶۴ بایت است). فایل‌های بزرگ مانند ویدئو یا تصاویر، تا زمانی که در محدودیت‌های واقعی زیر بگنجند، بدون مشکل ذخیره می‌شوند.
- محدودیت واقعی، **ظرفیت آزاد استخر** است — مجموع ظرفیت قابل‌استفاده‌ی دیسک‌های متصل، منهای سربار افزونگی هر سطح RAID. برای نمونه، با RAID-Z2 (توازن دوگانه)، محدودیت مؤثر تقریباً برابر با ظرفیت ترکیبی دیسک‌های داده است.
- یک فراخوانی خواندن/نوشتن WinFsp توسط خودِ API ویندوز به حدود ۴ گیبی‌بایت (`u32`) محدود می‌شود، اما این همان محدودیتی است که هر سیستم فایل واقعی دارد — سیستم‌عامل/برنامه به‌طور خودکار انتقال‌های بزرگ‌تر را به چند فراخوانی تقسیم می‌کند، بنابراین محدودیت عملی به‌شمار نمی‌رود.
- به دلیل کپی هنگام نوشتن، هر عملیات نوشتن (ایجاد، الحاق یا رونویسی) همواره به دست‌کم یک نوار (stripe) آزاد در استخر نیاز دارد (همان ایده‌ی `slop space` در ZFS). یک نوار دیگر نیز به‌طور دائم برای ذخیره‌ی فراداده رزرو شده است. پر کردن استخر تا ۱۰۰٪ ظرفیت به این معناست که حتی رونویسی داده‌های موجود نیز شکست می‌خورد. در عمل، همیشه چند درصد از استخر را آزاد نگه دارید.

## محدودیت‌های فعلی (مرحله‌ی نمونه‌ی اولیه)

- سوارکردن WinFsp فقط از یک فضای نام مسطح در ریشه پشتیبانی می‌کند. زیرپوشه‌ها پشتیبانی نمی‌شوند (ایجاد/حذف/تغییرنام برای هر فایل پشتیبانی می‌شود).
- خواندن/نوشتن از طریق `Pool::read_unaligned`/`Pool::write_unaligned_growing` (یک لایه‌ی خواندن-اصلاح-نوشتن) انجام می‌شود و از افست‌ها/طول‌های دلخواه بر حسب بایت پشتیبانی می‌کند؛ نوشتنی که از اندازه‌ی فعلی فراتر رود، فایل را به‌طور خودکار بزرگ‌تر می‌کند (برای ظرفیت و ملاحظات PATH به «ظرفیت و محدودیت‌های اندازه‌ی فایل» در بالا مراجعه کنید).
- `Pool` هم از `RaidZVdev` و هم از `Raid10Vdev` پشتیبانی می‌کند، اما یکپارچگی RAID10 با API مجموعه‌داده در برخی جاها هنوز سطحی است.
- کد سوارکردن واقعی WinFsp (`mount.rs`) را نمی‌توان با نسخه‌ای از ابزارهای Rust قدیمی‌تر از 1.85 ساخت، زیرا crateی `winfsp` به ویژگی Cargo به‌نام `edition2024` نیاز دارد (به بخش «ساخت و آزمایش» در زیر مراجعه کنید).
- `mount.rs` و پیاده‌سازی GPU در `zfs_accel_hlsl` (ویژگی `gpu`) به crateی `windows` وابسته‌اند که محتوای آن کاملاً خالی است مگر آنکه هدف کامپایل واقعاً ویندوز باشد. بنابراین این کد را فقط می‌توان روی یک دستگاه واقعی ویندوز ساخت و آزمایش کرد (یا هنگام کامپایل متقاطع برای هدف ویندوز)؛ روی Linux/macOS فقط پس از غیرفعال‌کردن این ویژگی‌ها از طریق `--no-default-features` ساخته می‌شود.
- تغییرنام (`rename`) یک فایل درحالی‌که هندل باز دیگری هنوز به آن اشاره دارد، می‌تواند باعث شود آن هندل دیگر برای عملیات‌های بعدی از کار بیفتد (`FileHandle` طبق طراحی نام را مستقیماً نگه می‌دارد — برای جزئیات به مستندات `Pool::rename_dataset` مراجعه کنید).

## ساخت و آزمایش

```powershell
cd open_runo_zfs_source/open_raid_z_core
cargo test --no-default-features   # بدون سوارکردن WinFsp یا شتاب‌دهی GPU (فقط منطق CPU؛ نه dxc و نه SDK ویندوز فسپ لازم است)
cargo test                         # پیش‌فرض (شامل سوارکردن واقعی WinFsp و شتاب‌دهی GPU/NPU؛ به WinFsp + dxc نیاز دارد)
```

`--no-default-features` هر دو ویژگی `winfsp_backend` و `gpu_accel` را غیرفعال می‌کند و امکان بررسی منطق اصلی — RAID0/1/5/6/10/Z2/Z3، چک‌سام‌های خودترمیم‌شونده، CoW، عکس‌های لحظه‌ای/کلون‌ها، resilver و مانند آن‌ها — را به‌صورت مستقل از سیستم‌عامل فراهم می‌کند (روی Linux/macOS نیز کار می‌کند). نه WinFsp، نه کامپایلر سایه‌ی DirectX (dxc) و نه سخت‌افزار GPU/NPU لازم نیستند.

ساخت با ویژگی‌های پیش‌فرض (`winfsp_backend` + `gpu_accel`) به موارد زیر نیاز دارد:

- نصب‌بودن زمان اجرای WinFsp (https://winfsp.dev/) روی سیستم (هدرهای SDK استفاده‌شده در زمان ساخت به‌طور خودکار همراه پروژه ارائه می‌شوند، بنابراین نیازی به نصب جداگانه‌ی مؤلفه‌ی توسعه‌دهنده نیست).
- در دسترس‌بودن `dxc` (کامپایلر سایه‌ی DirectX، همراه با Windows SDK یا Vulkan SDK) در `PATH` (برای کامپایل شیدرهای HLSL توازن RAID-Z/Z2 در زمان ساخت استفاده می‌شود).
- **نسخه‌ی Rust 1.85 یا جدیدتر** (نسخه‌ای که در آن `edition2024`، موردنیاز crateی `winfsp`، پایدار شده است؛ ابزارهای قدیمی‌تر حتی در تجزیه‌ی مانیفست `Cargo.toml` نیز شکست می‌خورند).

می‌توان WinFsp یا dxc را نیز به‌طور جداگانه غیرفعال کرد (برای نمونه `--no-default-features --features gpu_accel` برای فقط GPU، بدون WinFsp).

**نکته درباره‌ی اجرای واقعی آزمون‌های `winfsp_backend` (سوارکردن واقعی)**: crateی `winfsp` فایل DLL مربوط به WinFsp (`winfsp-x64.dll`) را به‌صورت پویا از طریق `LoadLibraryW` بارگذاری می‌کند که فقط در مسیر جستجوی استاندارد DLL (پوشه‌ی خودِ فایل اجرایی، `System32` و `PATH`) جستجو می‌کند. در محیط‌هایی که نصب‌کننده‌ی WinFsp خودش را به `PATH` اضافه نکرده باشد، ساخت به‌درستی انجام می‌شود (نیازی به هدرهای SDK ویندوز فسپ نیست)، اما اجرا **همیشه در زمان اجرا شکست می‌خورد** (خطای `WIN32(1285)` = `ERROR_DELAY_LOAD_FAILED`). پوشه‌ی `bin` مربوط به WinFsp را فقط برای اجرای آزمون‌ها به `PATH` اضافه کنید:

```powershell
$env:PATH = "C:\Program Files (x86)\WinFsp\bin;$env:PATH"
cargo test --features winfsp_backend,gpu_accel
```

بدون این کار، `mount_pool` مقدار `Err` برمی‌گرداند و آزمون آن را به‌عنوان مشکلی وابسته به محیط در نظر می‌گیرد، پیام رد شدن را از طریق `eprintln` چاپ می‌کند و زودتر بازمی‌گردد. **بدون `--nocapture`، این رد شدن همچنان فقط به‌صورت `ok` نمایش داده می‌شود و از موفقیت واقعی سوارکردن و خواندن/نوشتن قابل تشخیص نیست.** هنگام بررسی آزمون‌های سوارکردن واقعی، همیشه `--nocapture` را اضافه کنید و به‌صورت چشمی مطمئن شوید هیچ پیام رد شدنی نمایش داده نمی‌شود.

### ساخت و آزمایش نسخه‌ی لینوکس (FUSE)

```bash
# در اوبونتو/دبیان: به build-essential و pkg-config و libfuse3-dev نیاز است.
sudo apt-get install -y build-essential pkg-config libfuse3-dev

cd open_runo_zfs_source/open_raid_z_core
cargo test --no-default-features --features fuse_backend
```

ویژگی `fuse_backend` کریت `fuser` (اتصال واقعی به `libfuse3` در لینوکس) را فعال می‌کند. این ویژگی مستقل از `winfsp_backend`/`gpu_accel` است و در هدف‌های غیر لینوکسی قابل فعال‌سازی نیست، زیرا خودِ `fuser` حتی در آنجا یک وابستگی هم نیست (در `Cargo.toml` زیر `target.'cfg(target_os = "linux")'.dependencies` قرار دارد). آزمون یکپارچگی سوارکردن واقعی (`tests/fuse_mount.rs`) روی WSL2 Ubuntu 26.04 تأیید شده است: ایجاد، نوشتن، خواندن، تغییرنام، کوتاه‌کردن، حذف، رفت‌وبرگشت یک فایل بزرگ‌تر که چند نوار را دربرمی‌گیرد، و باقی‌ماندن فراداده پس از یک پیاده‌سازی و سوارکردن مجدد واقعی. اگر فقط روی ویندوز کار می‌کنید، WSL2 (‏`wsl --install`) روش توصیه‌شده برای ساخت/آزمایش هدف لینوکس است.

یک ابزار خط فرمان کوچک به نام `orzctl` نیز برای ایجاد و سوارکردن یک استخر مستقیماً از خط فرمان گنجانده شده است:

```bash
cargo build --no-default-features --features fuse_backend --bin orzctl
./target/debug/orzctl create --level z2 --chunk-size 4096 --stripes 1000 --dataset tank /path/to/disk0 /path/to/disk1 ...
./target/debug/orzctl mount  --level z2 --chunk-size 4096 --stripes 1000 --mountpoint /mnt/tank /path/to/disk0 /path/to/disk1 ...
```

برای سوارکردن خودکار هنگام بوت،
[`contrib/systemd/open-raid-z-pool.service.example`](../open_runo_zfs_source/open_raid_z_core/contrib/systemd/open-raid-z-pool.service.example)
را به‌عنوان یک واحد systemd ثبت کنید (روی یک VM VirtualBox تأیید شده:
استخری که روی ۴ دستگاه بلوکی واقعاً جدا از هم ایجاد شده، حتی پس از
یک راه‌اندازی مجدد واقعی نیز به‌طور خودکار سوار می‌شود).

### نصب‌کننده (`open_runo_installer` / `open_runo_installer_core`)

```powershell
# لایه‌ی منطق (بدون وابستگی به Tauri؛ روی Linux/macOS نیز اجرا می‌شود)
cd open_runo_zfs_source/open_runo_installer_core
cargo test                    # فقط بازگشت به CPU (پیش‌فرض)
cargo test --features gpu     # شامل دیسپچ واقعی GPU/NPU (به دستگاه واقعی ویندوز + dxc نیاز دارد)

# فرانت‌اند (TypeScript، مستقل از سیستم‌عامل)
cd open_runo_zfs_source/open_runo_installer
npm install
npx tsc --noEmit               # فقط بررسی نوع
npx vite build                 # ساخت واقعی

# خودِ برنامه‌ی Tauri (به دستگاه واقعی ویندوز، یا Rust به‌اندازه‌ی کافی جدید به‌همراه وابستگی‌های دسکتاپ لینوکس نیاز دارد)
cd open_runo_zfs_source/open_runo_installer/src-tauri
cargo tauri dev / cargo tauri build
```

`open_runo_installer_core` (شناسایی دیسک، مشاور پیکربندی به سبک Copilot، پیش‌نمایش مقداردهی اولیه‌ی zpool) یک crateی مستقل و بدون وابستگی به Tauri است، بنابراین منطق آن حتی در محیط‌هایی که فاقد نیازمندی‌های خودِ Tauri برای ساخت هستند (وب‌ویو، GTK و مانند آن‌ها، به‌همراه نسخه‌ای به‌اندازه‌ی کافی جدید از ابزارهای Rust) نیز قابل بررسی است. فقط شمارش واقعی دیسک‌ها (`\\.\PhysicalDriveN`) از یک API مخصوص ویندوز استفاده می‌کند و پشت `#[cfg(windows)]` جداسازی شده است؛ باقی موارد (مشاور پیکربندی و محاسبات پیش‌نمایش zpool) مستقل از سیستم‌عامل هستند و موفقیت هر ۲۶ آزمون آن‌ها تأیید شده است.

## نقشه راه: تعامل چند سیستم‌عامل و فرمت‌های موجود

هدف این پروژه این است که خودِ open-raid-z روی ویندوز، مک، لینوکس، اندروید و iOS/iPad نیز قابل خواندن و نوشتن باشد، و همچنین با فرمت‌های موجود سایر سیستم‌عامل‌ها (NTFS، exFAT، FAT32، ext4، APFS و مانند آن‌ها) تعامل‌پذیر باشد. امکان‌سنجی، اولویت‌بندی و محدودیت‌های فنی فعلی این مسیر — به‌ویژه این نکته که اپل اجازه‌ی پیاده‌سازی RAID روی دستگاه‌های بلوکی توسط اپلیکیشن‌های شخص ثالث را در iOS/iPad نمی‌دهد و به همین دلیل پشتیبانی از آن‌ها احتمالاً به مرور و مشاهده از طریق یک File Provider Extension محدود خواهد ماند — در فایل [`MULTIPLATFORM_ROADMAP.md`](open_runo_zfs_source/open_raid_z_core/contrib/systemd/MULTIPLATFORM_ROADMAP.md) ثبت شده است. شتاب‌دهی GPU/NPU نیز به‌تدریج در هر سیستم‌عامل که DirectX در آن در دسترس نیست، به سمت API بومی همان سیستم‌عامل حرکت خواهد کرد (برای نمونه، Metal Performance Shaders روی مک، NNAPI روی اندروید). همچنین تعامل‌پذیری با فرمت‌های RAID شخص ثالث (مانند mdadm در لینوکس، Storage Spaces در ویندوز) به‌عنوان یک هدف آینده در نظر گرفته شده است
