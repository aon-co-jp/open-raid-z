# open-raid-z

مشروع نظام ملفات تجريبي لنظامي Windows/Linux يحافظ على توافق شبه كامل مع NTFS/exFAT، مع توفير ميزات بأسلوب ZFS (مجاميع اختبارية ذاتية الإصلاح، مجمّعات تخزين، النسخ عند الكتابة، اللقطات/الاستنساخ) إلى جانب RAID0/1/5/6/10/Z2/Z3. المنطق الأساسي هو برنامج مشترك واحد مستقل عن نظام التشغيل (`open_raid_z_core`)؛ نسخة Windows (WinFsp) ونسخة Linux (FUSE) لا تختلفان إلا في طبقة التركيب الرقيقة فوقه (تُوزَّعان تحت اسمي `open-raid-z-win`/`open-raid-z-linux`).

## رسالة إلى Microsoft وApple

نعمل على تطوير نظام الملفات التجريبي هذا لجلب ميزات كاملة بأسلوب ZFS (مجاميع اختبارية ذاتية الإصلاح، RAID6/RAID-Z2، اللقطات، وغيرها) إلى Windows. أحد أهدافنا طويلة المدى هو أن يصبح نظام الملفات هذا يومًا ما خيارًا رسميًا للتثبيت وقرص إقلاع على Windows وmacOS.

ندرك أن هذا يتطلب تعاونًا من كل مزوّد نظام تشغيل — توقيع/اعتماد برامج تشغيل الإقلاع، والدعم الرسمي في برنامج التثبيت، وما إلى ذلك. إذا كان لديكم اهتمام بهذا المسعى، فسنرحب بشدة بتواصلكم وتعاونكم. هذا مشروع صغير ومستقل، لكننا نتمنى بصدق تحقيق هذه التقنية.

## قاعدة التسمية

المعرّفات التي يحدّدها هذا المشروع بنفسه — أسماء المجلدات، وأسماء الحزم (crates)، وأسماء حزم npm، وأسماء ميزات (features) Cargo، ومعرّفات/أصناف HTML/CSS، وما إلى ذلك — تستخدم بشكل متسق **الشرطة السفلية (`_`) بدلاً من الشرطة (`-`)** (مثل `open_raid_z_core`، و`zfs_accel_hlsl`، و`open_runo_installer`، و`open_runo_installer_core`، وميزات Cargo مثل `winfsp_backend`/`gpu_accel`). الأسماء التي كانت تحتوي سابقًا على شرطات، مثل `openzfs-winfsp-bridge`، أُعيدت تسميتها لضمان الاتساق داخل المشروع.

تُستثنى الحالات التالية لأنها تتبع مواصفات خارجية أو اصطلاحات النظام البيئي، وليس اختيار التسمية الخاص بهذا المشروع:

- اسم المستودع نفسه (`open-raid-z`؛ وهو الاسم الفعلي لمستودع GitHub ولا يمكن تغييره)
- سمات HTML5 المخصّصة من نوع `data-*` (مثل `data-i18n`؛ الشرطة إلزامية بحسب المواصفة)
- أسماء حزم npm الخارجية (مثل `@tauri-apps/api`، وهي الأسماء الفعلية للحزم المنشورة)
- أسماء خصائص CSS (مثل `font-family`؛ وهي صياغة لغة CSS نفسها)
- المصطلحات الإنجليزية المركّبة التي تحتوي فعليًا على شرطة، مثل Reed-Solomon أو copy-on-write

## المكوّنات

| المكوّن | الدور |
|---|---|
| `open_raid_z_core` | أجهزة vdev لـ RAID-Z/RAID0-10، مجمّع التخزين، طبقة توافق قوائم التحكم بالوصول NTFS/سمات exFAT، تركيب حقيقي (Windows = WinFsp عبر `mount.rs` / Linux = FUSE عبر `fuse_mount.rs`؛ كل شيء عدا طبقة التركيب الخاصة بكل نظام تشغيل مشترك بالكامل) |
| `zfs_accel_hlsl` | تفريغ حساب التكافؤ إلى مسرّعات عتادية GPU/NPU (DirectX 12 Compute + DirectML) |
| `open_runo_installer_core` | منطق مستقل عن نظام التشغيل لاكتشاف الأقراص، ومستشار التهيئة على طراز Copilot، ومعاينة تهيئة zpool (بلا أي اعتماد على Tauri؛ يعمل `cargo test` أيضًا على Linux/macOS) |
| `open_runo_installer` | المثبّت المبني على Tauri نفسه (طبقة واجهة رفيعة تستدعي `open_runo_installer_core`): اكتشاف العتاد، معالج تهيئة zpool، واجهة مستشار التهيئة على طراز Copilot |

## الميزات الرئيسية

- **دعم كامل لسلسلة RAID**: RAID0 / RAID1 (مرآة) / RAID5 / RAID6 / RAID10 (مرايا مقسّمة) / RAID-Z2 / RAID-Z3
- **تقسيم القرص وإعادة استخدامه**: تقسيم قرص فعلي واحد واستخدام نصفه كعضو في مرآة بينما ينضم النصف الآخر إلى مصفوفة RAID6/Z2 منفصلة
- **مجاميع اختبارية ذاتية الإصلاح، النسخ عند الكتابة، اللقطات/الاستنساخ**: تحاكي أسلوب ZFS. يمكن لـ`Pool::scrub` اكتشاف الأعطال الصامتة عبر المجمّع بأكمله وإصلاحها في مرور واحد، باستخدام نفس واجهة البرمجة سواء لعائلة RAID-Z أو RAID10
- **التوافق مع NTFS**: ترجمة قوائم التحكم بالوصول (NFSv4 ⇔ NTFS)، وربط UID/GID ⇔ SID (ربط حتمي قائم على RID لنطاقات SAM المحلية/AD)
- **التوافق مع exFAT**: تحويل سمات الملفات والطوابع الزمنية، ودعم الملفات/الوحدات التخزينية التي تتجاوز 4 جيجابايت
- **تسريع عتادي عبر GPU/NPU**: يتم فعليًا إرسال حساب تكافؤ RAID-Z1/Z2 عبر DirectX 12 Compute + DirectML (يعود تلقائيًا إلى المعالج عند عدم وجود عتاد)
- **مستشار تهيئة على طراز Copilot**: يوصي بمستوى RAID بناءً على تخطيط الأقراص والمسرّع وعدد أنوية المعالج (نسخة أولى قائمة على قواعد استدلالية؛ كما يتوفر هيكل أساسي لاكتشاف نموذج لغوي محلي). يوجد المنطق في `open_runo_installer_core`، المستقل عن Tauri، ويمكن التحقق منه أيضًا باستخدام `cargo test` على Linux/macOS
- **تركيب حقيقي عبر WinFsp (‏Windows)**: يمكن تركيبه فعليًا كحرف محرك أقراص في Windows. تظهر كل مجموعة بيانات في المجمّع كملف مستقل خاص بها، مع دعم إزاحات بايت عشوائية للقراءة والكتابة، بالإضافة إلى إنشاء الملفات وحذفها وإعادة تسميتها والإلحاق بها واقتطاعها (لا تزال مساحة الأسماء مسطحة في الجذر — المجلدات الفرعية غير مدعومة بعد). تم التحقق منه على عتاد حقيقي: القراءة والكتابة والإنشاء والحذف وإعادة التسمية والإلحاق والاقتطاع للملفات عبر محرك أقراص مركَّب فعليًا.
- **تركيب حقيقي عبر FUSE (‏Linux)**: يمكن تركيب نفس `Pool` مباشرةً على Linux أيضًا (`fuse_mount.rs`)، بنفس وظائف نسخة Windows (إنشاء/حذف/إعادة تسمية/إلحاق/اقتطاع). تم التحقق منه من طرف إلى طرف على WSL2 Ubuntu 26.04 — تم تركيبه واختباره عبر نداءات `std::fs` العادية. ولأنه قائم على inode، فهو لا يشترك في القيد المعروف لنسخة WinFsp حيث قد يستمر مقبض آخر مفتوح بالإشارة إلى اسم قديم بعد إعادة التسمية. كما تحتوي حزمة `fuser` على ميزة `macfuse-4-compat`، لذا فمن المحتمل أن يمتد نفس التصميم مستقبلًا إلى macOS (كوحدة بيانات، وليس كقرص إقلاع).
- **دعم متعدد اللغات**: يستخدم المثبّت اليابانية افتراضيًا مع مبدّل لغة في الواجهة، قابل للتغيير أيضًا بعد التثبيت
- **أداة ترحيل البيانات الموجودة (وحدة `migrate`، تجريبية)**: تنسخ شجرة مجلدات NTFS (أو ما شابهها) موجودة إلى المجمّع. لا تكتب أبدًا في المصدر، لذا يمكن تشغيلها **بينما يستمر تشغيل Windows**. لكنها لا يمكنها تحويل **قرص النظام الجاري تشغيله حاليًا (C: وغيره) في مكانه** إلى تنسيق RAID دون توقف (لا يمكن لنظام التشغيل أن يسمح بإعادة كتابة القرص الذي يستخدمه فعليًا بواسطة برنامج يعمل عليه هو نفسه — هذا قيد أساسي، وليس ميزة ناقصة). إنها أداة "نسخ إلى مكان آخر (المجمّع)" على وجه الحصر. حاليًا هي دالة مكتبة فقط، بلا واجهة سطر أوامر أو رسومية بعد؛ يتم تسطيح المجلدات الفرعية إلى مستوى واحد باستخدام حرف فاصل.
- **ديمومة البيانات الوصفية (‏`Pool::save`/`Pool::open`)**: يمكن الآن حفظ قائمة مجموعات البيانات وتخصيصات الشرائح واللقطات وغيرها من معلومات الإدارة في منطقة محجوزة (كتلة فائقة) داخل المجمّع واستعادتها منها. لم تكن هذه الآلية موجودة سابقًا — كانت بايتات البيانات الخام تبقى على القرص، لكن سجل مكان كل ملف كان يُفقد بمجرد إنهاء العملية (التركيب). تحفظ الآن كل من نسخة Windows (‏WinFsp) ونسخة Linux (‏FUSE) تلقائيًا عند كل عملية تعديل، وتم التحقق على عتاد حقيقي من أن الملفات تبقى فعلًا بعد إلغاء تركيب حقيقي وإعادة تركيب.

## السعة وحدود حجم الملفات

- يُتتبَّع الحجم المنطقي لمجموعة البيانات (الملف) باستمرار كنوع `u64`، لذا لا يوجد حد اصطناعي مثل حاجز 4 جيجابايت في FAT32 (الحد النظري هو 2^64 بايت). الملفات الكبيرة مثل الفيديو أو الصور تُحفظ دون مشاكل ما دامت ضمن القيود الفعلية أدناه.
- الحد الفعلي هو **السعة الحرة للمجمّع** — مجموع السعة القابلة للاستخدام للأقراص المتصلة، مطروحًا منه النفقات الإضافية للتكرار الخاصة بكل مستوى RAID. على سبيل المثال، مع RAID-Z2 (تكافؤ مزدوج)، يكون الحد الفعلي تقريبًا هو السعة المجمّعة لأقراص البيانات.
- تقتصر عملية قراءة/كتابة واحدة عبر WinFsp على نحو 4 جيجابايت (`u32`) بحسب واجهة برمجة Windows نفسها، لكن هذا هو نفس القيد الموجود في أي نظام ملفات حقيقي — إذ يقوم نظام التشغيل/التطبيق تلقائيًا بتقسيم عمليات النقل الأكبر إلى عدة نداءات، لذا لا يشكّل قيدًا عمليًا.
- بسبب النسخ عند الكتابة، تحتاج كل عملية كتابة (سواء إنشاء أو إلحاق أو استبدال) دائمًا إلى شريحة واحدة حرة على الأقل متاحة في المجمّع (نفس فكرة `slop space` في ZFS). كما تُحجز شريحة إضافية بشكل دائم لتخزين البيانات الوصفية. ملء المجمّع إلى 100% من سعته يعني أن حتى استبدال البيانات الموجودة سيفشل. من الناحية العملية، يُنصح دائمًا بترك بضع نسب مئوية من المجمّع حرّة.

## القيود الحالية (مرحلة النموذج الأولي)

- يدعم تركيب WinFsp فقط مساحة أسماء مسطحة في الجذر. المجلدات الفرعية غير مدعومة (أما إنشاء/حذف/إعادة تسمية الملفات الفردية فمدعومة).
- تمر عمليات القراءة/الكتابة عبر `Pool::read_unaligned`/`Pool::write_unaligned_growing` (طبقة قراءة-تعديل-كتابة) وتدعم إزاحات/أطوال بايت عشوائية؛ وتؤدي الكتابة التي تتجاوز الحجم الحالي إلى توسيع الملف تلقائيًا (انظر «السعة وحدود حجم الملفات» أعلاه بخصوص السعة واعتبارات PATH).
- يدعم `Pool` كلاً من `RaidZVdev` و`Raid10Vdev`، لكن تكامل RAID10 مع واجهة برمجة مجموعة البيانات لا يزال سطحيًا في بعض الأجزاء.
- لا يمكن بناء شيفرة التركيب الحقيقي عبر WinFsp (`mount.rs`) بإصدار من أدوات Rust أقدم من 1.85، لأن حزمة `winfsp` تتطلّب ميزة Cargo المسمّاة `edition2024` (انظر قسم البناء والاختبار أدناه).
- يعتمد كل من `mount.rs` وتنفيذ GPU في `zfs_accel_hlsl` (ميزة `gpu`) على حزمة `windows`، التي يكون محتواها فارغًا تمامًا ما لم تكن منصّة الترجمة فعليًا نظام Windows. لذلك لا يمكن بناء هذه الشيفرة واختبارها إلا على جهاز Windows حقيقي (أو عند الترجمة المتقاطعة إلى منصّة Windows)؛ وعلى Linux/macOS، تُبنى فقط بعد تعطيل هذه الميزات عبر `--no-default-features`.
- قد يؤدي إعادة تسمية (`rename`) ملف بينما لا يزال مقبض آخر مفتوح يشير إليه إلى تعطّل ذلك المقبض الآخر في العمليات اللاحقة (يحتفظ `FileHandle` بالاسم مباشرة بحسب التصميم — راجع توثيق `Pool::rename_dataset` للتفاصيل).

## البناء والاختبار

```powershell
cd open_runo_zfs_source/open_raid_z_core
cargo test --no-default-features   # بدون تركيب WinFsp أو تسريع GPU (منطق المعالج فقط؛ لا حاجة إلى dxc أو حزمة تطوير WinFsp)
cargo test                         # الوضع الافتراضي (يشمل التركيب الحقيقي عبر WinFsp وتسريع GPU/NPU؛ يتطلب WinFsp + dxc)
```

يعطّل `--no-default-features` كلتا الميزتين `winfsp_backend` و`gpu_accel`، مما يتيح التحقق من المنطق الأساسي — RAID0/1/5/6/10/Z2/Z3، والمجاميع الاختبارية ذاتية الإصلاح، والنسخ عند الكتابة، واللقطات/الاستنساخ، وresilver، وغيرها — بشكل مستقل عن نظام التشغيل (يعمل أيضًا على Linux/macOS). لا حاجة إلى WinFsp أو مترجم ظلال DirectX (dxc) أو عتاد GPU/NPU.

يتطلّب البناء بالميزات الافتراضية (`winfsp_backend` + `gpu_accel`):

- تثبيت بيئة تشغيل WinFsp (https://winfsp.dev/) على النظام (يتم تضمين ملفات ترويسة SDK المستخدمة أثناء البناء تلقائيًا، لذا لا حاجة لتثبيت منفصل لمكوّن المطوّرين).
- توفّر `dxc` (مترجم ظلال DirectX، المرفق مع Windows SDK أو Vulkan SDK) ضمن `PATH` (يُستخدم لترجمة ظلال HLSL الخاصة بتكافؤ RAID-Z/Z2 أثناء البناء).
- **إصدار Rust 1.85 أو أحدث** (الإصدار الذي استقرّت فيه ميزة `edition2024` التي تتطلبها حزمة `winfsp`؛ الإصدارات الأقدم تفشل حتى في تحليل ملف البيان `Cargo.toml`).

يمكن أيضًا تعطيل WinFsp أو dxc كل على حدة (مثل `--no-default-features --features gpu_accel` لتفعيل GPU فقط، بدون WinFsp).

**ملاحظة عند تشغيل اختبارات `winfsp_backend` فعليًا (التركيب الحقيقي)**: تُحمِّل حزمة `winfsp` ملف WinFsp الديناميكي (`winfsp-x64.dll`) ديناميكيًا عبر `LoadLibraryW`، الذي يبحث فقط في مسار البحث القياسي عن ملفات DLL (مجلد الملف التنفيذي، و`System32`، و`PATH`). في البيئات التي لم يُضِف فيها مثبّت WinFsp نفسه إلى `PATH`، يكتمل البناء بنجاح (لا حاجة إلى ترويسات SDK الخاصة بـ WinFsp)، لكن التشغيل **يفشل دائمًا وقت التنفيذ** (الخطأ `WIN32(1285)` = `ERROR_DELAY_LOAD_FAILED`). أضف مجلد `bin` الخاص بـ WinFsp إلى `PATH` فقط عند تشغيل الاختبارات:

```powershell
$env:PATH = "C:\Program Files (x86)\WinFsp\bin;$env:PATH"
cargo test --features winfsp_backend,gpu_accel
```

بدون ذلك، تُعيد `mount_pool` قيمة `Err`، ويتعامل الاختبار مع الأمر كمشكلة تعتمد على البيئة، فيطبع رسالة تخطٍّ عبر `eprintln` ويعود مبكرًا. **بدون `--nocapture`، يظهر هذا التخطي مع ذلك كـ`ok` فقط، ولا يمكن تمييزه عن نجاح فعلي للتركيب والقراءة/الكتابة.** استخدم دائمًا `--nocapture` عند التحقق من اختبارات التركيب الحقيقي، وتأكّد بصريًا من عدم ظهور رسالة تخطٍّ.

### بناء واختبار نسخة Linux (‏FUSE)

```bash
# على Ubuntu/Debian: يلزم build-essential وpkg-config وlibfuse3-dev.
sudo apt-get install -y build-essential pkg-config libfuse3-dev

cd open_runo_zfs_source/open_raid_z_core
cargo test --no-default-features --features fuse_backend
```

تُفعّل ميزة `fuse_backend` حزمة `fuser` (ربط حقيقي بـ`libfuse3` في Linux). وهي مستقلة عن `winfsp_backend`/`gpu_accel`، ولا يمكن تفعيلها على أهداف غير Linux لأن `fuser` نفسها ليست حتى تبعية هناك (توجد ضمن `target.'cfg(target_os = "linux")'.dependencies` في `Cargo.toml`). تم التحقق من اختبار التكامل للتركيب الحقيقي (`tests/fuse_mount.rs`) على WSL2 Ubuntu 26.04 — إنشاء وكتابة وقراءة وإعادة تسمية واقتطاع وحذف، بالإضافة إلى تبادل ملف أكبر يمتد عبر عدة شرائح. إذا كنت تعمل على Windows فقط، يُنصح باستخدام WSL2 (‏`wsl --install`) لبناء واختبار هدف Linux.

```powershell
# طبقة المنطق (بلا اعتماد على Tauri؛ تعمل أيضًا على Linux/macOS)

# الواجهة الأمامية (TypeScript، مستقلة عن نظام التشغيل)
cd open_runo_zfs_source/open_runo_installer
npm install
npx tsc --noEmit               # فحص الأنواع فقط
npx vite build                 # بناء فعلي

# تطبيق Tauri نفسه (يتطلب جهاز Windows حقيقي، أو إصدار Rust حديث بما يكفي إضافة إلى تبعيات سطح مكتب Linux)
cd open_runo_zfs_source/open_runo_installer/src-tauri
cargo tauri dev / cargo tauri build
```

`open_runo_installer_core` (اكتشاف الأقراص، مستشار التهيئة على طراز Copilot، معاينة تهيئة zpool) هو حزمة مستقلة بلا اعتماد على Tauri، لذا يمكن التحقق من منطقها كما هو حتى في البيئات التي تفتقر إلى ما يحتاجه Tauri نفسه للبناء (WebView وGTK وغيرها، إضافة إلى إصدار حديث بما يكفي من أدوات Rust). فقط التعداد الفعلي للأقراص (`\\.\PhysicalDriveN`) يستخدم واجهة برمجة خاصة بـ Windows، وهي معزولة خلف `#[cfg(windows)]`؛ أما كل ما عداه (مستشار التهيئة وحسابات معاينة zpool) فمستقل عن نظام التشغيل، وقد تم التأكد من نجاح جميع اختباراته الـ26.
